一、问题的起点：为什么需要自动化乐评？



在撰写大量乐评的过程中，我发现自己反复在做同一件事：搜索专辑信息、整理曲目列表、分析音乐风格。如果能将这些重复性工作自动化，我就能把更多精力投入到真正的创作中——对音乐的深度解读和文学性表达。



于是，我设定了这样一个目标：在 Dify 中创建一个工作流，输入乐队和专辑名称，自动生成一篇具有文学深度和专业批评视角的乐评。



二、技术路线图：从简单到复杂的演进



第一阶段：直接调用 LLM（失败了）



最初的想法很简单：让 DeepSeek 直接根据专辑名称写乐评。结果发现，模型对较新专辑的曲目信息完全不了解，甚至会编造不存在的歌曲名称。



核心教训：大语言模型的知识是有限的，尤其是对最新发行的专辑，必须引入外部信息源。



第二阶段：引入 Google 搜索（部分成功）



我在工作流中加入了 Google 搜索节点，让模型基于搜索结果撰写乐评。这确实解决了信息缺失的问题，但很快发现了新问题：



Google 搜索返回的是 JSON 结构，LLM 无法直接解析。 搜索结果摘要（snippet）也常常被截断，无法获取完整曲目列表。



第三阶段：代码节点成为关键



我意识到，需要在搜索节点和 LLM 节点之间增加一个代码节点，负责数据转换和提取。



\`\``python

def main(arg1):

\# 从 Google 搜索 JSON 中提取关键信息

\    organic = arg1.get("organic_results", \[])

\    if organic:

\    snippet = organic\[0].get("snippet", "")

\    return {"context_text": f"曲目信息：{snippet}"}

\    return {"context_text": "未找到信息"}

\`\``



这个小小的代码节点，成为了整个工作流的“翻译官”。



第四阶段：SSL 问题的意外出现



当我尝试接入更专业的 MusicBrainz 音乐数据库时，遇到了一个意想不到的问题：



\`\``

SSLEOFError: EOF occurred in violation of protocol

\`\``



这个错误反复出现，让我意识到：Dify 部署环境的网络出口与某些境外 API 的 SSL 连接不稳定。这不是代码问题，而是基础设施层面的挑战。



三、解决方案：多层降级策略



最终，我设计了一个多层信息获取策略：



层级 数据来源 优势 适用场景

第一层 本地曲目库 零网络依赖，100%准确 常用专辑

第二层 MusicBrainz API 免费、开源、权威 主流专辑

第三层 Google 搜索 覆盖面广 冷门/最新专辑

第四层 LLM 自身知识 即时可用 经典专辑



\`\``python

def main(album_name, artist_name=""):

\# 第一层：本地库

\    if album_name in LOCAL_TRACKS:

\    return LOCAL_TRACKS\[album_name]

\    

\# 第二层：MusicBrainz

\    try:

\    tracks = fetch_from_musicbrainz(album_name, artist_name)

\    if tracks:

\    return tracks

\    except SSLException:

\    pass  # 降级

\    

\# 第三层：Google 搜索

\    return fetch_from_google(album_name)

\`\``



四、关键技术洞察



1. LLM 与结构化数据的鸿沟



大语言模型擅长处理自然语言，但无法直接理解 JSON、XML 等结构化数据。代码节点是连接两者的桥梁——将结构化数据转换为自然语言，再交给 LLM 处理。



2. 网络环境的重要性



在 Dify 这样的应用开发平台中，代码执行环境（通常是云端容器）的网络连通性直接影响着工作流的稳定性。对于需要访问境外 API 的场景，建议：



· 增加重试机制

· 配置超时时间

· 设计降级方案



3. 知识库 vs. 实时搜索



· 知识库：适合稳定的、不常变化的信息，响应快、成本低

· 实时搜索：适合最新信息，但依赖外部 API、有延迟成本



对于乐评场景，两者结合是最优解。



五、最终工作流架构



\`\``

开始节点（输入专辑、艺人）

\    ↓

┌───────────────────────────────────┐

│        代码节点（信息获取）         │

│  ┌─────────────────────────────┐  │

│  │ 本地曲目库 → MusicBrainz →  │  │

│  │ Google搜索 → LLM补全        │  │

│  └─────────────────────────────┘  │

└───────────────────────────────────┘

\    ↓

LLM 节点（生成乐评）

\    ↓

结束节点（输出结果）

\`\``



六、学习总结



通过这次实践，我学到了：



1. 复杂任务需要分层设计：将“生成乐评”拆分为“信息获取”和“文本生成”两个独立环节

2. 代码节点是 Dify 中最强大的工具：它可以完成任何 API 调用、数据转换、逻辑判断

3. 网络问题比代码问题更难排查：SSL 错误、超时、重试机制，这些都需要在代码层面优雅处理

4. 降级策略是工程化的核心：不要假设任何外部服务永远可用

5. LLM 不是万能的：它需要被喂养正确的、结构化的信息，才能产出高质量的内容



写在最后



从最初简单的“让 AI 写乐评”，到最终构建出一个多层降级、容错完善的智能工作流，这个过程让我深刻理解了：真正的 AI 应用，不是把任务丢给模型，而是为模型构建一个可靠的信息获取和处理管道。



如果你也在 Dify 中构建类似的 AI 工作流，希望这份记录能为你节省一些踩坑的时间。毕竟，技术探索的意义不在于“我做到了”，而在于“我知道为什么能做到，以及下一次如何做得更好”。
